🚀 1. Redirect
👉 Redirect changes the URL in browser.

Browser is told:
“Go to another URL”

Example
User opens:
/old-dashboard

Redirect sends user to:
/dashboard

Browser URL becomes:
/dashboard

⚡ Use cases
old route changed
login redirect
moved pages

Example
async redirects() {
  return [
    {
      source: "/old",
      destination: "/new",
      permanent: true,
    },
  ];
}

⚡ Use cases
proxy backend APIs
hide internal URLs
clean URLs

🚀 Production-Ready next.config.js
/** @type {import('next').NextConfig} */

const nextConfig = {
  async rewrites() {
    return [
      {
        source: "/api/:path*",
        destination:
          process.env.NODE_ENV === "development"
            ? "http://localhost:8080/:path*"
            : "https://api.healthtrack.com/:path*",
      },
    ];
  },
};

module.exports = nextConfig;

🔥 What happens
In local development

Frontend request:

/api/users

Internally goes to:

http://localhost:8080/users
In production

Same frontend request:

/api/users

Internally goes to:

https://api.healthtrack.com/users


🔥 Real production architecture
Browser
   ↓
Next.js frontend
   ↓
Rewrite/Proxy
   ↓
Spring Boot APIs

🧠 Case 1 — Internal Next.js API Route Exists

If you have:

pages/api/users.js

OR modern:

app/api/users/route.ts

THEN:

/api/users

will hit THIS internal Next.js API first.

👉 Rewrite will NOT happen for that matching route.

🧠 Case 2 — No Internal API Route Exists

Then rewrite rule can proxy request externally.

Example:

/api/users

internally rewritten to:

http://localhost:8080/users

Next.js first checks:

Does internal route exist?

If YES:
👉 use internal API route.

If NO:
👉 rewrite/proxy may apply.

**In genral, in production, routing/proxying is often handled by infrastructure layers instead.**

🧠 In enterprise architecture

Usually request flow is:

Browser
 ↓
Application Gateway / Load Balancer / Nginx / Ingress
 ↓
Frontend
 ↓
Backend APIs

So rewrite/proxy happens OUTSIDE Next.js.

### 🧠 yarn.lock

👉 yarn.lock locks the EXACT versions of installed packages.

Used by:
👉 Yarn

🔥 Why needed?

Without lock file:

Today install may get:

react 18.2.0

Tomorrow may get:

react 18.3.0

👉 Can cause bugs/inconsistency.

⚡ yarn.lock solves this

It ensures:
✅ Same package versions
✅ Same dependency tree
✅ Same install for everyone

🧠 Simple analogy
package.json

Says:

“I need React version 18”

yarn.lock

Says:

“Use EXACTLY this specific downloaded version”

🔥 Similar file in npm
Package Manager	Lock File
Yarn	yarn.lock
npm	package-lock.json
⚠️ Important

Usually:
✅ Commit lock file to Git
❌ Don’t edit manually

🚀 In one line

👉 yarn.lock stores exact dependency versions to ensure consistent installs across environments.

🧠 Example package.json
"react": "^18.2.0"

Notice:

^

This symbol matters A LOT.

⚡ What ^18.2.0 means

It means:

👉 Install:

18.2.0 OR compatible newer versions

So npm/yarn may install:

18.2.0
18.2.1
18.3.0

depending on availability.

It has 1 folder API, & 2 files _app.js and index.js

Anything inside API foler, vercel will make it server. All API related code will go inside it.
🧠 pages/index.js

👉 Main home page.
This file:
pages/index.js
maps to:
/
(home route)

🧠 pages/api
👉 Backend API routes inside Next.js.

Example:
pages/api/users.js
becomes:
/api/users

You can write backend logic here.
Example
export default function handler(req, res) {
  res.status(200).json({ name: "Ankit" });
}

/api/users does NOT change page/UI route.

Because:
👉 It is an API endpoint, not a frontend page.

🧠 Difference
Route Type	        Purpose
pages/index.js	    UI page
pages/api/users.js	Backend API

👉 Acts like ROOT component for all pages.

Used for:

global layouts
providers
global CSS
shared wrappers

export default function App({ Component, pageProps }) {
  return <Component {...pageProps} />
}
🧠 Think like this

Whenever page changes:

_app.js
   ↓
Current Page
⚡ Common use cases of _app.js

Authentication wrapper

🔥 Mapping Summary
File/Folder	    Purpose
pages/index.js	Home page (/)

👉 pages/ = older routing system where:

index.js = homepage
